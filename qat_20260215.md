## 前提

- 差分基準は `origin/main` ではなく、`https://github.com/huggingface/sentence-transformers/pull/3655` の head (`c00b9a5acd83199f1891da2e2e6bedeb6899edcd`)。
- 取得確認には `gh pr view 3655 --repo huggingface/sentence-transformers` を使用。
- 4パターン:
  - Baseline (MNRL): `mpnet-base-gooaq-qat-trainb1m-r1-baseline-mnrl-bs128-20260214-092545`
  - TrainB Control: `mpnet-base-gooaq-qat-trainb1m-r1-trainb-control-bs128-20260214-102012`
  - Step4b Ref: `mpnet-base-gooaq-qat-trainb1m-r1-step4b-bs128-20260214-104545`
  - Warmup1800: `mpnet-base-gooaq-qat-trainb1m-r1-warmup1800-bs128-20260214-111124`

## 実験設定（4パターン共通）

- 評価: `nanobeir`（`msmarco,nq`）
- 評価量子化モード: `legacy`
- eval時クエリ量子化: `true`
- binary 復元: `zero_one`
- int8 range strategy: `minmax`
- 学習サンプル: `1,000,000` / epoch: `1.0` / batch size: `128` / seed: `42`

## 1. TrainB Control と PR #3655 の差分

### 学習（train）側

- 実験用スクリプト追加:
  - `examples/sentence_transformer/training/quantization/train_qat_gooaq_ablation.py`
  - `examples/sentence_transformer/training/quantization/run_trainb_research_rounds_configurable.sh`
- PR #3655 の `train_qat_gooaq.py` は固定設定だが、TrainB Control は引数で明示的に以下を指定:
  - `--train-loss qat`
  - `--train-binary-mode signed`
  - `--no-train-use-int8-range-state`
  - `--train-quantization-warmup-steps 0`
- `QuantizationAwareLoss` 実装差分（PR #3655 基準）:
  - `binary_mode` が追加され、`binary` を `signed(-1/+1)` で扱える。
  - int8 range state（EMA）機能が追加されたが、TrainB Control では `use_int8_range_state=false` で無効化。
  - quantization warmup 機能が追加されたが、TrainB Control では `0` で無効化。
  - forward cache 取り出し時に `.copy()` して、量子化処理で float32 キャッシュを破壊しないように修正。

### 評価（eval）側

- PR #3655 にはない evaluator 拡張が追加されている（`InformationRetrievalEvaluator`）。
- ただし TrainB Control は `eval_quantization_mode=legacy` のため、calibration/dequantize 戦略は実質未使用（PR挙動に近い）。
- 差分として効く可能性がある点:
  - `binary` 展開時のオーバーフロー回避（`astype(np.int16) + 128`）の安全化。
- さらに PR #3655 の example が GooAQ dev 評価中心なのに対し、TrainB Control は NanoBEIR 評価を標準化している。

## 2. Step4b Ref と TrainB Control の差分

### 学習（train）側

- TrainB Control からの変更点:
  - `train_use_int8_range_state: false -> true`
  - `train_precision_warmup_steps: null -> [0,200,1200]`
  - `train_int8_range_momentum: 0.99`（明示指定）
  - `train_quantization_warmup_steps: 0`（据え置き）
- 影響する実装:
  - `QuantizationAwareLoss.resolve_warmup_steps_by_precision(...)`
  - `_scaled_weight(...)` による precision 別線形 warmup
  - int8 range EMA (`use_int8_range_state`, `int8_range_momentum`)

### 評価（eval）側

- TrainB Control から変更なし（`legacy`, `zero_one`, `quantize_queries=true` など同一）。

## 3. Warmup1800 と TrainB Control の差分

### 学習（train）側

- TrainB Control からの変更点:
  - `train_use_int8_range_state: false -> true`
  - `train_precision_warmup_steps: null -> [0,200,1800]`
- Step4b Ref との差分は、binary warmup のみ:
  - Step4b Ref: `[0,200,1200]`
  - Warmup1800: `[0,200,1800]`

## 4. Baseline (MNRL) の位置づけ（PR #3655 基準）

- `train_loss=mnrl` のため QAT loss を使わない対照実験。
- `train_binary_mode`, `train_use_int8_range_state`, `train_quantization_warmup_steps` などは設定値としては保存されるが、`MNRL` 実行では実質無効。
- つまり Baseline は「PR #3655 由来の QAT 経路」ではなく、同じデータ分割・同じ eval 条件での非QAT比較軸。

## 5. QAT設定だけで見た PR #3655 と TrainB の差

### 結論

- QAT設定だけ見れば、`TrainB Control` は `PR #3655` にかなり近い。
- TrainB Control は「PR #3655 相当の挙動を、拡張実装上で明示フラグ指定して再現した設定」。

### 比較（QAT設定）

| 項目 | PR #3655 | TrainB Control |
| --- | --- | --- |
| train loss | QAT (`QuantizationAwareLoss`) | QAT (`--train-loss qat`) |
| precisions | `float32,int8,binary` | `float32,int8,binary` |
| weights | `1.0,1.0,0.5` | `1.0,1.0,0.5` |
| binary mode | 実装上ほぼ固定（signed相当） | `--train-binary-mode signed` |
| int8 range EMA | なし | 機能はあるが `--no-train-use-int8-range-state` でOFF |
| precision別 warmup | なし | 機能はあるが `--train-quantization-warmup-steps 0` で実質OFF |

### 参考: Baseline (MNRL)

- `--train-loss mnrl` のため、QAT経路そのものを使わない。
- よって QAT関連オプション（binary_mode / int8 range state / warmup）は、設定値が残っていても学習ロジックには効かない。

## 6. 追加実装の要点（今回の会話で確認した3点）

### `eval_quantization_mode=legacy` とは

- 旧来の評価経路を使うモード。
- `model.encode(..., precision=...)` の結果を evaluator 側で float に戻して類似度計算する。
- evaluator 拡張の calibration/dequantize 戦略は基本的に使わない。

### cache copy fix とは

- 対象: `QuantizationAwareLoss` 内の forward cache 再利用。
- 修正: `output = self.cache[self.idx]` ではなく `output = self.cache[self.idx].copy()` を使用。
- 意図: 量子化処理が float32 キャッシュ本体を破壊して、次precisionに汚染が波及するのを防ぐ。

### evaluator拡張の存在とは

- `InformationRetrievalEvaluator` に以下の拡張を追加:
  - `quantization_eval_mode` (`legacy` / `evaluator`)
  - calibrationサイズとrange戦略（`minmax` / `rolling_std`）
  - `quantization_dequantize`
  - `binary_reconstruction` (`zero_one` / `minus_one_one`)
  - `quantize_queries`
- なお今回の4パターンは `legacy` 評価なので、これらの新機能の多くは比較軸に入っていない。

## 7. 実務上の読み方

- `PR #3655 vs TrainB Control` の純QAT比較をしたい場合:
  - QAT設定差は小さい前提で見る。
- それでも結果差がある場合:
  - 評価セット差（GooAQ dev vs NanoBEIR）、seed、サンプル数、batch size、実装安全化（cache copy fix等）を先に疑う。
